**Version:** 1.0

# 1. Purpose

This document defines the deployment architecture, infrastructure, containerization strategy, CI/CD pipeline, monitoring, and production operations for Sentinel AI.

Deployment goals:

* One-command local setup
* Automated CI/CD
* Reproducible builds
* Zero manual server configuration
* Cloud portability

---

# 2. Deployment Philosophy

Sentinel follows:

* Infrastructure as Code
* Immutable deployments
* Container-first architecture
* Environment-based configuration
* Automated releases
* Stateless application servers

# 5. Local Development

Requirements

* Docker Desktop
* Git
* Python 3.12
* Node.js 22
* pnpm

Clone repository

```bash id="w2x8sl"
git clone https://github.com/<user>/sentinel.git
cd sentinel
```

Start services

```bash id="r6a0pm"
docker compose up --build
```

Available Services

| Service     | Port |
| ----------- | ---- |
| Frontend    | 3000 |
| Backend API | 8000 |
| PostgreSQL  | 5432 |
| Redis       | 6379 |
| ChromaDB    | 8001 |

# 9. Build Strategy

Backend

* Multi-stage Docker build
* Slim Python image
* Install only production dependencies

Frontend

* Build static assets
* Standalone Next.js output

Goals

* Small images
* Fast startup
* Cached dependencies

---

# 10. CI Pipeline

Every Pull Request executes:

1. Install dependencies
2. Lint
3. Format check
4. Type checking
5. Unit tests
6. Integration tests
7. Frontend tests
8. Security scan
9. Build Docker images

Pipeline fails immediately on any critical error.

---

# 11. CD Pipeline

On merge to `main`:

1. Build production images
2. Push images
3. Apply database migrations
4. Deploy backend
5. Deploy frontend
6. Restart workers
7. Run smoke tests
8. Notify deployment status

# 19. Backup Strategy

PostgreSQL

Daily backups

Retention

30 days

ChromaDB

Weekly snapshot

Repository clones

Never persisted after scans complete.

---

# 20. Rollback Strategy

If deployment fails:

1. Stop rollout
2. Restore previous container image
3. Reconnect previous database schema
4. Validate health
5. Notify maintainers

No manual code edits in production.

---

# 21. Scaling Strategy

Horizontal

* Backend API
* Celery workers

Vertical

* PostgreSQL
* ChromaDB

Queue-based architecture enables worker scaling without changing application logic.

---

# 22. Security Hardening

Production requirements

* HTTPS only
* Secure headers
* Rate limiting
* Input validation
* Non-root containers
* Read-only filesystem where possible
* Automatic dependency updates

---

Versioning

v0.2.0

v1.0.0

Release checklist

* CI passes
* Tests pass
* Security scan passes
* Documentation updated
* Changelog generated
* Version tag created

---

# 24. Disaster Recovery

Recover by:

* Restoring PostgreSQL backup
* Restoring ChromaDB snapshot
* Redeploying containers
* Running health checks
* Validating scan workflow

Target Recovery Time (RTO)

<30 minutes

Target Data Loss (RPO)

<24 hours
