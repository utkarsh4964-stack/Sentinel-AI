# 1. Purpose

This document defines every security rule supported by Sentinel AI.

It specifies:

* Supported vulnerability classes
* Detection strategy
* Detection tools
* AI reasoning policy
* Severity mapping
* Confidence scoring
* False positive handling
* Future extensibility

This document is the authoritative reference for all security analysis performed by Sentinel AI.

---

# 2. Security Philosophy

Sentinel follows an **Evidence-First Security Model**.

Security findings originate only from:

* Static analysis tools
* Dependency scanners
* Secret scanners
* Repository metadata
* Repository architecture

LLMs **never** create vulnerabilities.

LLMs only:

* Explain findings
* Prioritize findings
* Recommend remediation
* Estimate business impact

# 4. Supported Severity Levels

| Severity | Description                                       | SLA Recommendation        |
| -------- | ------------------------------------------------- | ------------------------- |
| Critical | Immediate exploitation or complete compromise     | Fix immediately           |
| High     | Serious vulnerability with practical exploitation | Fix within 24 hours       |
| Medium   | Security weakness with limited impact             | Fix during current sprint |
| Low      | Minor issue or hardening recommendation           | Fix when convenient       |
| Info     | Informational finding                             | No immediate action       |

# 8. Confidence Scoring

Confidence is assigned after normalization.

| Score    | Meaning                     |
| -------- | --------------------------- |
| 95–100   | Confirmed by multiple tools |
| 80–94    | High confidence             |
| 60–79    | Medium confidence           |
| 40–59    | Low confidence              |
| Below 40 | Advisory only               |

LLMs do not increase confidence.

They may reduce confidence if evidence is incomplete.

---

# 9. False Positive Strategy

Every finding receives one of:

* Confirmed
* Likely
* Possible
* Needs Review

Users may mark findings as:

* Ignored
* Accepted Risk
* Resolved

Ignored findings remain in history.

---

# 11. Evidence Collection

Every finding must include:

* Affected file
* Line number
* Rule ID
* Triggering code snippet
* Scanner output
* Related symbols
* Repository context

LLM explanations must reference only collected evidence.

---

# 12. AI Reasoning Rules

The AI must never:

* Invent vulnerabilities
* Change severity without evidence
* Recommend unsafe fixes
* Execute repository code
* Assume business context

The AI may:

* Explain findings
* Prioritize remediation
* Describe exploit scenarios
* Suggest secure coding practices

---

# 13. Risk Scoring Formula

Overall Repository Score

100

− Critical Findings Penalty

− High Findings Penalty

− Medium Findings Penalty

− Secret Exposure Penalty

− Dependency Risk Penalty

Final score is clamped between 0 and 100.

# 14. Rule Update Strategy

Security rules are versioned.

Rule updates must:

* Preserve backward compatibility
* Include migration notes
* Be independently testable

Rule packs are loaded dynamically.

# 16. Security Constraints

Sentinel must never:

* Execute repository code
* Upload repositories to unknown services
* Modify source code without user approval
* Store secrets permanently
* Log sensitive credentials

---

# 17. Testing Strategy

Every rule must include:

* Positive test cases
* Negative test cases
* Edge cases
* False positive validation
* Performance benchmarks

New rules cannot be released without automated regression tests.
